# CLAUDE.md — JeDAI 일반상식 KB 데이터 구축 전용 프로젝트

> 이 문서는 신규 프로젝트(별도 폴더/레포, JeDAI 3개 레포와 무관)의 진입점으로 쓰기 위한 초안이다.
> 새 프로젝트 루트에 `CLAUDE.md`로 두고 Claude Code 세션을 시작하면 된다.

## 프로젝트 목적 (단일 목적)

JeDAI 채팅의 "전역 일반상식 KB"(`jedai_general_knowledge` Qdrant 컬렉션)에 관리자가 업로드할
**CSV 데이터셋을 웹 검색 기반으로 구축·검증·병합**하는 것. 이 프로젝트의 산출물은 오직
**CSV 파일 1개**다. 코드를 배포하거나 JeDAI 시스템에 편입되는 것이 아니라, 개방망(인터넷 연결)
개발 머신에서 실행해 결과물만 JeDAI 관리자 페이지(`/admin` → 일반 상식 → CSV 업로드)에 수동
업로드하는 **데이터 준비 도구**다.

**하지 않는 것**: JeDAI 코드 수정, Qdrant/Postgres 직접 접근, 임베딩 생성(임베딩은 JeDAI AI 서버가
업로드 시점에 사내 bge-m3로 수행 — 이 프로젝트는 평문 CSV만 만든다).

## 배경 — 이 데이터가 왜 정확해야 하는가

JeDAI는 사용자 질문마다 이 KB를 조회해 관련도가 높으면 **"모델 자체 학습지식보다 우선하는
공식 기준 정보"**로 시스템 프롬프트에 주입한다(침묵 그라운딩, tool 호출이 아니라 매 턴 무조건
실행). 즉 여기 잘못된 행이 들어가면 사내 폐쇄망 LLM이 **확신을 갖고 틀리게** 답한다.
따라서 이 프로젝트의 제1원칙은 **양보다 정확도** — 검증 안 되는 사실은 만들지 않는다.

## 산출물 계약 — CSV 스키마 (JeDAI 업로드 요구사항과 100% 일치해야 함)

```
statement,category,effective_date,source
```

| 컬럼 | 규칙 |
|---|---|
| `statement` | 검색 임베딩 대상인 유일한 컬럼. **한 행 = 하나의 완전히 독립된 사실**. 청킹 없이 이 문장 그대로 임베딩되므로 대명사·생략 없이 문장 하나만 읽어도 완결되게 쓴다. 쉼표 포함 시 큰따옴표로 감싼다. 셀 내부 줄바꿈 금지. |
| `category` | 아래 고정 taxonomy 중 정확히 하나. 임의 신규 카테고리 생성 금지. |
| `effective_date` | 그 사실이 성립/확인된 기준일, `YYYY-MM-DD`. |
| `source` | 실제 출처(기관명·매체명 구체적으로). 불명확하면 그 행 자체를 버린다. |

- 헤더 행 필수, UTF-8.
- 이 CSV는 JeDAI 업로드 시 **전체 교체(replace)** 방식으로 처리된다 — 이 프로젝트에서는 여러 번
  나눠 올리는 게 아니라, 모든 배치를 **로컬에서 병합한 최종 파일 1개**를 만드는 게 목표다.

## 카테고리 taxonomy & 포함 기준

| 카테고리 | 포함 여부 | 규칙 |
|---|---|---|
| `인사` | 자유롭게 포함 | 공개된 주요 지휘관·정부 인사 현직 정보(합참의장·각군 참모총장·국방부 장관/차관 등). 최우선 카테고리. |
| `조직` | 자유롭게 포함 | 부대 창설/해체/통합, 사령부 개편, 직제 변경 등 공식 발표 사실. |
| `정책` | 자유롭게 포함 | 국방개혁·병역제도 등 공식 발표된 제도 변경사항만(추진 중/검토 중 단계 제외). |
| `시사` | 자유롭게 포함 | 대통령·국무총리 등 범용 시사 인물/직책 정보. |
| `전력화` | 자유롭게 포함 | 공식 보도자료로 확정 발표된 신규 전력·장비 도입 현황("전력화 완료" 등 확정 문구 기준). |
| `정세` | 제한적 포함 | **공신력 있는 공식 발표 스냅샷만.** 문장 안에 발표 시점을 명시("OO일 기준 국방부 발표에 따르면..."). 정세 분석·전망 문장 금지. |
| `일정` | 제한적 포함 | 이미 공식 발표된 확정 일정만. 유동적 계획 단계 정보 금지. 지나간 일정이 남아 오답 유발하지 않도록 재검토 주기 필요(아래 "유지보수" 참고). |
| `전쟁사실` | 제한적 포함 | **이미 종결되어 더는 바뀌지 않는 역사적 사건만**(예: "2022-02-24 러시아가 우크라이나를 침공했다"). 현재 진행 중인 전황·전세·최근 전투 결과·"누가 우세한가" 판단은 **절대 포함 금지** — 변동이 가장 빠르고 소스마다 상충되는 영역이라 재업로드 주기로 못 따라감. |
| `경제지표` | 제한적 포함 | 한국은행·통계청·IMF 등 **공식 발표 수치 스냅샷만**("2026년 6월 기준 기준금리는 X%, 출처: 한국은행"). "경기가 좋아지고 있다" 류 정성적 추세 서술 금지. |

**빼는 것**: 군사기밀·비공개 인사정보·미발표 계획 등 비공개 정보 전반. 이 KB는 공개된
사실(공식 언론보도·정부 발표 기준)만 다룬다.

## 사실 검증 프로토콜 (모든 행 공통)

1. **웹 검색으로 실시간 확인한 사실만** 작성한다. 학습 데이터 기억만으로 작성 금지.
2. 수치·날짜·고유명사가 명확히 박힌 **확정된 사실**만 작성한다. 판단·전망·추세성 서술
   ("~일 것으로 보인다", "회복세다", "우세하다" 등)은 생성하지 않는다.
3. 출처가 웹 검색으로 명확히 확인 안 되면 그 행은 만들지 말고 건너뛴다. 개수를 채우려고
   불확실한 내용을 넣지 않는다.
4. 카테고리별 추가 규칙(위 표)을 반드시 함께 적용한다.

## 실행 워크플로 (Claude Code 관점)

이 프로젝트는 외부 채팅 UI에 프롬프트를 복붙하는 방식이 아니라, **Claude Code 세션이 직접
WebSearch 도구로 검색·검증하며 CSV 행을 작성**하는 방식으로 진행한다.

### 1) 카테고리별 배치 생성
- 카테고리 1개씩, 배치당 30~50행 목표(품질 우선 — 검증 안 되면 그보다 적어도 됨).
- 카테고리가 서로 독립적이므로 **Agent 툴로 카테고리별 서브에이전트를 병렬 실행**하는 것을
  권장한다(예: `인사`/`조직`/`정책`/`시사`/`전력화`를 동시에 리서치). 단, `정세`/`전쟁사실`/
  `경제지표`처럼 규칙이 까다로운 카테고리는 규칙을 놓치기 쉬우니 단독으로 신중히 실행하거나
  결과를 별도로 한 번 더 검토한다.
- 각 서브에이전트에게는 위 "산출물 계약 + 카테고리별 규칙 + 검증 프로토콜"을 그대로 프롬프트로
  전달하고, WebSearch 도구 사용을 명시적으로 지시한다.

### 2) 중복 방지
- `seen.md`(또는 `seen.json`) 파일에 지금까지 생성된 statement의 핵심 주제(예: "합참의장-현직",
  "기준금리-2026-06")를 누적 기록한다.
- 다음 배치 프롬프트에 이 목록을 포함시켜 "이미 다룬 항목과 사실상 같은 내용은 다시 만들지
  말라"고 명시한다.
- 배치 완료마다 `seen.md`를 갱신한다(이 갱신은 Claude Code가 매 배치 후 자동으로 수행).

### 3) 로컬 후처리 (병합·중복 제거)
- 모든 배치 CSV를 하나로 병합한다.
- 완전 동일 문자열 중복은 물론, 표현만 다른 유사 중복(같은 인물의 같은 직책, 같은 지표의 같은
  시점 등)까지 검토한다 — 기계적 dedup(정확 일치)은 스크립트로, 의미적 유사 중복은 마지막에
  Claude Code가 한 번 훑어보며 판단한다.
- 헤더·컬럼 개수·quoting이 깨진 행이 없는지 최종 검증(파이썬 `csv` 모듈로 파싱해보는 정도의
  기계적 검증 권장).

### 4) 산출
- 최종 파일: `output/general_knowledge_YYYYMMDD.csv` (병합·중복제거 완료본 하나).
- 이 파일을 JeDAI 관리자 페이지(일반 상식 → CSV 업로드)에서 업로드하면 전체 교체된다.

## 프로젝트 폴더 구조 (제안)

```
jedai-general-knowledge-builder/
├── CLAUDE.md                  # 이 문서
├── seen.md                    # 누적 "이미 다룬 주제" 목록 (중복 방지용)
├── batches/                   # 카테고리·회차별 원본 배치 CSV (중간 산출물, 보존)
│   ├── 인사_01.csv
│   ├── 정세_01.csv
│   └── ...
└── output/
    └── general_knowledge_YYYYMMDD.csv   # 최종 병합본 (업로드 대상)
```

## 업로드 전 QA 체크리스트

- [ ] 헤더가 정확히 `statement,category,effective_date,source`인가
- [ ] 모든 행의 `category`가 taxonomy 9종 안에 있는가(오탈자·신규 카테고리 없는가)
- [ ] `effective_date`가 전부 `YYYY-MM-DD` 형식인가
- [ ] `source`가 비어있거나 "확인 안 됨" 상태인 행이 없는가
- [ ] `전쟁사실` 카테고리에 현재진행형 전황/전세 판단 문장이 섞여있지 않은가
- [ ] `정세`/`경제지표`에 정성적 추세 서술(판단·전망)이 섞여있지 않은가
- [ ] 완전 중복 또는 의미상 중복 행이 없는가
- [ ] CSV가 파이썬 `csv` 모듈로 에러 없이 파싱되는가(컬럼 개수 불일치 행 없는가)

## 유지보수 주기 (참고)

카테고리별로 재업로드(재생성) 필요 주기가 다르다. 참고용 권장치 — 실제 운영 주기는 KB
관리 담당자가 정한다.

| 카테고리 | 권장 재검토 주기 |
|---|---|
| `인사`, `조직` | 인사 발령·개편 발표 시점마다 (비정기, 발생 즉시) |
| `정세`, `일정` | 짧게(주 단위) — 안 지키면 지난 일정/스냅샷이 최신인 것처럼 남는 위험 큼 |
| `경제지표` | 월 단위(지표 발표 주기에 맞춤) |
| `전쟁사실` | 사실상 정적 — 신규 종결 사건 발생 시에만 |
| `정책`, `전력화`, `시사` | 비정기, 공식 발표 시점마다 |

이 표는 JeDAI 쪽 코드에 자동 만료 로직이 없다는 전제(현재 `general_kb.py`는 전체교체만 지원,
`effective_date`는 메타데이터일 뿐 자동 필터링에 쓰이지 않음)를 반영한 것 — 재검토는 이
빌더 프로젝트를 다시 실행하는 사람의 운영 책임이다.
